# Plan: Close the Creator OS learning loop (audit steps 1+2+4)
_Locked via grill — by Claude + Ader. Rev 6 (post codex Rounds 1–5)._

## Goal
Turn the currently-empty feedback loop into a flowing one: every new Creator OS post carries complete lineage, the hourly ThreadsDash sync fans real outcomes out to all three factories (Campaign scoring, Reel winner-DNA, Reference outcomes) in one idempotent job, and human taste labels (Gold/Maybe/Ignore) soft-weight reference ranking. Exit criterion: the `closed_loop_learning_status` readiness gate counts real forward-only posts toward its 50-post / 1h+24h threshold, with no manual outcome imports required. Explicitly deferred: bulk generation, experiments framework, autonomy expansion (audit steps 3, 5, 6).

## Approach (steps ordered; step 0 is a hard prerequisite)

0. **Fix metric-history retrieval 400 + history provenance — lands first.**
   Reproduce the failing request in `campaign_factory/adapters/threadsdash.py` (`_select_threadsdash_post_metric_history`), diagnose, fix, add a regression test with the captured failing request shape.
   **Fallback rows must not poison the gate.** Today, when history retrieval fails — or succeeds but omits a post — `_threadsdash_performance_rows` substitutes the ordinary post row, which imports as metrics-eligible (threadsdash.py:4153, :4662), and readiness derives 1h/24h windows from `snapshot_at - published_at` (learning_readiness.py:15), so fallback rows masquerade as real history. Fix: tag every imported snapshot with `history_source` (`metric_history` | `post_row_fallback`). Fallback rows are excluded from BOTH the readiness gate and fan-out. Sync stays fail-open (must not die) but the report shows fallback counts per run with error detail.

1. **Complete lineage at the real write points; lineage contract v2 + draft-payload contract v2 (dual-validation migration).**
   Current contract `generated_asset_lineage.v1.schema.json` only requires `source`/`generation`/`review` objects — `referenceId`/`promptId` are optional, yet `import_prompt_outcomes` requires at least one (outcomes.py:23). So v1 conformance cannot guarantee Reference fan-out works. Ship **lineage v2**: for pipeline-generated drafts, require **`promptId`** (not `referenceId`-alone — a reference can hold multiple prompts, `reference_factory/db.py:319` UNIQUE(reference_id, target_tool, model_profile), so reference-only records cannot be resolved to one prompt and would strand fan-out as permanently-pending; `referenceId` stays as a redundant cross-check field) plus campaign/recipe/caption/rendered-asset identifiers. **Variant identity is conditional**: the normal draft path defaults to no variation (`adapters/threadsdash.py:63` `enable_variation: bool = False`) and `rendered_assets.variant_id` is nullable (db.py:177) — v2 requires `variantId` only when the draft went through the variation stage; base-asset drafts carry `variantId: null` and their stable identity is the rendered-asset ID. For varied drafts, lineage `variantId` is stamped from `variantAssignment.variant_asset_id` (produced in variation_stage.py:238) in the adapter **before** final v2 validation — NOT from `publishability.variant_id`, which threadsdash.py:3142 sources from base-asset publishability and which stores `variant_asset_id` separately; the assignment is the authoritative variant identity. **Lineage-v2 conformance is itself part of learning eligibility**: a post-cutover post whose stored draft lacks valid v2 lineage (e.g. ingested as v1 during the migration window) is `metrics_eligible` for raw storage but fails `learning_eligible` — fenced out of the gate and fan-out just like manual posts, counted in the sync report. **Identifiers that are legitimately resolved downstream of export are conditional, not hard-required**: `audio_id` may be null at export when native audio selection happens at publish time (threadsdash.py:3037 resolves it from audio_selection/handoff at publish). `audio_intent.v1` defines **no ID field** and `_build_audio_intent` (threadsdash.py:2689) emits none — so lineage v2 requires an **`audio_intent_fingerprint`**: a deterministic hash (canonical-JSON SHA-256, algorithm documented in the schema) over the intent payload's semantic fields. **Computed in the adapter, not exports.py**: shared lineage is assembled at exports.py:203, but the final destination-specific audio intent is built later in `adapters/threadsdash.py` — so the fingerprint is stamped onto each final draft's lineage immediately after `_build_audio_intent` returns, and the v2 hard gate validates at that point. No change to `audio_intent.v1` itself; platform `audio_id` is an optional late-bound field the sync backfills once known.
   **No in-place mutation of a pinned v1 contract; consumers dual-accept BEFORE producers flip.** `campaign_draft_payload.v1.schema.json` stays as-is. Add `campaign_draft_payload.v2.schema.json` referencing lineage v2. **All three validation surfaces** learn v2: the Python validator (validator.py:18 pins v1 filename), the hand-written TypeScript validator (`packages/pipeline_contracts/typescript/index.ts:12,1253` — hard-codes v1 and is NOT touched by schema regeneration; needs manual v2 support), and the ThreadsDashboard ingest that calls it (`../ThreadsDashboard/api/_lib/handlers/campaign-factory/draftIngest.ts:439` — must be deployed with dual-accept). Strict sequencing: (a) ship dual-accept in Python + TS validators, (b) deploy ThreadsDashboard, (c) only then flip exports.py to emit v2, (d) remove v1 acceptance after all producers emit v2. Schema mirrors/examples regenerated for v2 only.
   Enforcement points (actual lineage writers): `campaign_factory/exports.py` (final draft export — the hard gate) plus upstream populators `account_planning.py`, `creative_planning.py`, `motion_edit_stage.py`.
   - Pipeline-generated drafts (through exports.py): **hard block** — export fails loudly on incomplete v2 lineage.
   - Manual posts: **warn + leave existing `metrics_eligible=0`** (column exists, campaign db.py:364, DEFAULT 0 — no new migration). Post proceeds, excluded from the gate, leak surfaced in the sync report.

2. **Forward-only counting via explicit cutover — one shared eligibility predicate for ALL learning readers.**
   Add config `LEARNING_LOOP_CUTOVER` (ISO timestamp, set at deploy). Define **one predicate** `learning_eligible(snapshot)` in `learning_score.py`: `metrics_eligible=1` AND `history_source='metric_history'` AND `published_at` non-null AND `published_at >= LEARNING_LOOP_CUTOVER` AND `lineage_v2_valid=1` (stored draft passes lineage-v2 validation; stamped at snapshot write, backfilled by the migration validator — this clause appears in **both** the Python callable and the shared SQL fragment, so a post-cutover v1-lineage record fails the predicate itself, not just prose) (null `published_at` ⇒ ineligible, reported). This predicate governs **every learning input** — the complete enumerated consumer list, all of which currently query `metrics_eligible=1` (or all snapshots) directly: `learning_readiness.py` (gate), `performance_summary.py:34` (Campaign scoring) **and** its asset-planning scores (`performance_summary.py:180` via `account_planning.py:331`), `account_reward_baselines` inputs (learning_score.py:17), `recommendations.py:400` (pattern/preset recommendations) **and** its measurement path (`measure_recommendation_item` / proved-disproved marking, recommendations.py:2095, :2775), `account_memory.py:41` (account memory rebuild), `creative_knowledge.py:1211`, `winner_expansion.py:403`, `variant_lineage.py:1129` (`variant_metrics_rollup`, which feeds `creator_os_recommendations.py:128` variant-generation recommendations), and bridge fan-out. Implementation: a shared SQL fragment/helper next to the predicate so no consumer hand-rolls the filter; each consumer gets a test proving pre-cutover/fallback rows are excluded. `metrics_eligible` alone remains a raw-storage flag, not a learning-eligibility signal. No backfill of the 952 historic posts.

3. **Bridge = fan-out phase inside the existing hourly sync (no new job).**
   Campaign snapshot storage is an **upsert** on `(post_id, snapshot_at)` that overwrites all metric fields (threadsdash.py:4246, db.py:448) — not append-only. The bridge accounts for that:
   **The script's existing unledgered Reel refresh is replaced, not duplicated.** `scripts/sync_threadsdash_performance.py:88` currently runs `build_refresh_command` (Reel `metrics_store` refresh over ALL `metrics_eligible=1` snapshots, metrics_store.py:555) after every sync — that path bypasses cutover/provenance and would double-write alongside the ledger. The fan-out phase **subsumes it**: the standalone refresh invocation is removed from the script, and `metrics_store` ingestion is called only from the ledger-controlled bridge, filtered by the step-2 predicate.
   **Ledger**: table keyed **`(post_id, snapshot_at, destination)`** (exact timestamp — no hour rounding, so multiple snapshots per hour can't collide) with columns: `status` (`pending`/`done`/`superseded`/`failed_capped`/`retracted`), `attempt_count`, `source_hash` (see below), **`destination_record_id`** (the concrete written identity: `(prompt_id, post_id)` for Reference, `outcome_id` for Reel), `scoring_version` + `baseline_provenance` (reference destination only), `last_error`.
   - **Identity-changing corrections replace transactionally.** Because the ledger records the exact destination identity it wrote, a correction that changes attribution (prompt P1→P2, or a different Reel `outcome_id`) is handled as: write new identity + retract old identity in one transaction — never write-new-and-leave-old.
   - **Retraction recomputes, never blind-deletes.** When several snapshots collapse into one destination fact, retracting one snapshot triggers recompute of that fact from the **newest remaining eligible snapshot** (restoring prior data if the newest was retracted; keeping newer data if an older one was).
   - **`publish_metrics` is ledger-controlled too.** `metrics_store` also upserts `publish_metrics` keyed by filename, unconditionally (metrics_store.py:700) — that write goes through the same ledger/monotonic-guard path as `reel_outcomes`, not around it. Because `publish_metrics` is keyed by filename while `outcome_id` also encodes account+posted_at (metrics_store.py:652), distinct outcomes can collapse onto one filename row: the ledger therefore records **both** `outcome_id` and the filename identity per destination write, and on retraction/correction the collapsed `publish_metrics` row is recomputed from the newest remaining eligible outcome sharing that filename (deleted if none remain).
   - **`source_hash` = hash of the complete normalized fan-out input, computed per destination** — not just raw metric fields. It covers everything the destination write depends on: raw metrics, account identity, lineage IDs (reference/prompt/campaign/recipe/caption/variant/rendered-asset), rendered output path, `history_source` provenance, and eligibility. The Campaign upsert (threadsdash.py:4246) can correct any of these without touching metrics; a metrics-only hash would leave `done` rows preserving wrong attribution.
   - **Change detection**: on each sync, if a destination row's recomputed `source_hash` differs from the ledger's, that row reopens to `pending` with `attempt_count` reset — corrected upstream data re-propagates.
   - **Retraction on eligible→ineligible**: the bridge scans **all previously-ledgered rows regardless of current eligibility**, not just currently-eligible snapshots. If a ledgered snapshot no longer satisfies `learning_eligible` (e.g. corrected to `metrics_eligible=0`, or provenance downgraded), its destination facts are explicitly retracted: the `prompt_post_outcomes` row is deleted and the prompt aggregate recomputed; the Reel outcome row is deleted (or zeroed with a retraction note); the ledger row is marked `retracted` (new terminal status alongside `superseded`/`failed_capped`). Dedicated eligible→ineligible test.
   - **State semantics**: `done` only when the destination write returns a verified written/updated identity. Soft-skips are NOT done: `import_prompt_outcomes` reports missing matches as `skipped` (outcomes.py:84) and Reel bridge failures like missing output paths are soft (metrics_store.py:587) — both leave the row `pending` with `last_error` and `attempt_count += 1`, retried next run. When `attempt_count` reaches the cap (config, default 5) the row moves to terminal `failed_capped` with a loud alert; it re-enters `pending` only via a `source_hash` change or manual reset. A write intentionally rejected by the stale-snapshot guard (below) is marked `superseded` — terminal and correct, never retried, never counted as a failure.
   - **Reference outcomes become per-post facts, aggregated deterministically.** The current model is one overwrite-in-place scalar per prompt row (`generated_video_prompts.outcome_*`, reference db.py:304) and `import_prompt_outcomes` can match by `reference_id` alone, updating every prompt for a reference (outcomes.py:50) — a newer post would erase earlier posts' evidence. Fix: new table `prompt_post_outcomes` keyed `(prompt_id, post_id)` storing the stamped scalar + `source_snapshot_at` + baseline provenance per post; per-post upsert requires a resolved `prompt_id` (fan-out that resolves only to `reference_id` is a `skipped`/pending condition, never a broadcast update). The prompt-level `outcome_*` columns become a **derived aggregate** (recency-weighted mean, `outcome_sample_count` = number of distinct posts) recomputed after each per-post upsert.
   - **Reference read paths are denormalized — fan-out must propagate to them.** Prompt outcomes reach ranking only when `analyze_patterns` embeds them into `reference_patterns.pattern_json` (patterns.py:50), which `learning.py:151` then reads from stored JSON; `build_learning_system` does not refresh existing patterns by default (learning.py:33 — only when `refresh_patterns` or pattern count is low). Without propagation, hourly fan-out would never move performance classes or cluster ranking. Fix: after the Reference destination phase, the bridge triggers a **targeted pattern refresh for exactly the references whose prompt aggregates changed this run** (re-running the `measuredOutcome` embedding for those pattern rows — not a full `analyze_patterns` pass). End-to-end test: new snapshot → fan-out → prompt aggregate → pattern_json → changed `_performance_class`/cluster order. Stale-snapshot guard now applies per `(prompt_id, post_id)`: an incoming `source_snapshot_at` < stored value ⇒ `superseded`.
   Destinations per snapshot:
   - **Campaign**: existing upsert path (unchanged semantics, now hash-tracked).
   - **Reel**: feed raw metrics to `intelligence_store`. Its "latest-wins" behavior today comes only from the `MAX(snapshot_at)` source query (metrics_store.py:569) — the destination `reel_outcomes` upsert overwrites unconditionally and stores no source snapshot (intelligence_store.py:70, metrics_store.py:649). Under per-snapshot ledger retries a delayed older snapshot could overwrite newer metrics, so add a `source_snapshot_at` column to `reel_outcomes` (additive migration) with the same monotonic guard as Reference: overwrite only when incoming `source_snapshot_at` >= stored; older ⇒ ledger row `superseded`. **Raw metrics only** — Reel winner-DNA keeps its own logarithmic `winner_score` (intelligence_store.py:339, winner_dna.py:925); the bridge does not inject rewardScore into Reel.
   - **Reference**: auto-generate records in the exact `import_prompt_outcomes` shape, reward stamped per step 4.

4. **rewardScore: one named per-snapshot callable; frozen-baseline stamps for Reference.**
   `account_normalized_decay_shrinkage.v1` is a version string returned by cohort-level `learning_summary` (learning_score.py:115), not a per-snapshot function — so extract one with an **exact equation**: **`snapshot_normalized_reward(snapshot, baselines) -> float`** in `learning_score.py` returns the **unshrunk relative reward**: `snapshot_reward(snapshot) / max(0.000001, baseline)` where `baseline` = account median from `account_reward_baselines` or `DEFAULT_REWARD_BASELINE` prior — i.e., exactly the `relative_reward` intermediate `learning_summary` already computes per snapshot (learning_score.py:147ff). NOT the 0–100 `score` (would classify nearly everything as `performed_well`) and NOT shrunk (shrinkage is a cohort-level decay-weighted aggregate; applying it per snapshot would silently change the meaning of the 1.05/0.85 thresholds). Parity target: for any snapshot, the callable's value equals the `relative_reward` computed inside `learning_summary` for the same snapshot+baselines — asserted by test. Consumer map (exhaustive):
   - **Reference fan-out (write)**: scalar stamped at fan-out via this callable.
   - **Reference read paths**: `patterns.py` `_performance_class` (1.05/0.85 thresholds) consumes stamped rewardScore — same scale, thresholds untouched.
   - **Campaign read paths**: `learning_summary`/readiness compute from raw at read time via the same callable.
   - **Reel**: NOT a consumer — `winner_score` is independent by design (see step 3).
   **Baseline drift**: account baselines are recomputed medians of current latest snapshots (learning_score.py:17), so new posts silently shift older normalized rewards. Stamped Reference outcomes therefore **freeze their baseline**: each stamp records `baseline_provenance` (account, median value, sample n, computed_at) and is immutable for that `(scoring_version, baseline_provenance)`. Recalibration is explicit only: bump `scoring_version` ⇒ reference ledger rows reopen ⇒ re-stamp with fresh baselines. No silent drift, no perpetual churn.

5. **Gold/Maybe/Ignore as soft weights — wired where cluster ranking actually lives.**
   Cluster ranking is implemented in `reference_factory/learning.py` (:234, :481), and `_pattern_cards` (learning.py:151) currently doesn't load `review_labels` — so the work is: join `review_labels` into the learning.py ranking path and apply config-driven multipliers (Gold ≈1.5x, Maybe ≈0.9x, Ignore ≈0.3x, unlabeled 1.0x) to a reference's cluster-ranking weight there. Explicitly NOT applied to `rewardScore` or `_performance_class` — label weights never reclassify measured performance. Reason capture: existing `tags` param on `label_reference` (review.py).

6. **Observability.** Sync report gains: posts scanned / imported / ineligible (with reasons incl. null published_at, manual-no-lineage), fallback-row counts (`history_source`), fan-out per destination: done / pending / reopened-by-hash / retry-capped, readiness-gate progress (n/50 with 1h+24h coverage, cutover shown).

7. **Tests.** Regression test for the 400 fix; `learning_eligible` predicate tests **per enumerated consumer** (gate, performance_summary + asset planning, baselines, recommendations, account_memory, creative_knowledge, winner_expansion, fan-out) proving pre-cutover/fallback/null-published rows are excluded; lineage v2 validation (hard-block vs warn, null-audio-with-fingerprint accepted, fingerprint determinism, base-asset draft with `variantId: null` passes, varied draft has `variantId == variantAssignment.variant_asset_id` and passes (positive variation-stage test), varied draft whose `variantId` was sourced from publishability instead of the assignment fails, promptId-missing fails, post-cutover v1-lineage post fenced from `learning_eligible`); draft-payload dual-validation across **both** Python and TypeScript validators (v1 and v2 accepted by readers, exports emit v2 only after dual-accept ships); ledger tests (double-run no-op, partial-failure retry with attempt_count, retry cap → `failed_capped`, source_hash reopen on non-metric attribution change, two-snapshots-same-hour, soft-skip stays pending, stale snapshot → `superseded`, **eligible→ineligible → `retracted` with destination facts recomputed from newest remaining eligible snapshot**, identity-change P1→P2 replaces transactionally without orphaning P1, `publish_metrics` writes guarded); per-post Reference outcome tests (two posts on one prompt both retained, aggregate recomputed, reference-only match never broadcasts); Reel monotonic-guard test (delayed older snapshot cannot overwrite newer `reel_outcomes` row); end-to-end Reference propagation test (snapshot → fan-out → prompt aggregate → targeted pattern refresh → changed performance class/cluster order); `snapshot_normalized_reward` parity test (callable == `learning_summary`'s internal `relative_reward` for identical inputs; stamped value classifies identically through `_performance_class`); script-level test that the standalone Reel refresh is no longer invoked outside the ledger; learning.py test proving labels alter actual cluster order while performance classes are unchanged.

## Key decisions & tradeoffs
- **Scope = audit steps 1+2+4 only** (option B): bridge is required for the 50-post cohort to self-measure; steps 3/5/6 deferred until a first cohort exists.
- **Forward-only, no backfill** (Q2-A): fuzzy-matching 952 historic posts would poison the training data. Cost: gate starts at 0/50. Enforced by explicit cutover + provenance, not flag semantics alone.
- **Soft weights, no hard Ignore exclusion** (Q3-B, user override of Claude's rec): Gold set unproven (225 labels, 205 captioned vs 20 visual, zero outcome validation). Configurable so they can harden later.
- **Bridge inside hourly sync** (Q4-A): sync owns the ThreadsDash session, schedule, dedupe; a second job is a second failure point with no benefit.
- **Frozen-baseline stamps over live recompute for Reference** (Q5-C, revised R2): live baselines drift as cohorts grow; frozen stamps + explicit version bumps make every stored scalar reproducible and auditable.
- **Split lineage enforcement at exports.py** (Q6-C): hard block on the pipeline path doubles as a bug detector; manual posts stay frictionless but visibly ineligible.
- **Reel scoring independence**: winner-DNA's log winner_score is a different objective (visual pattern mining, not account-normalized outcome); unifying scales is not needed to close the loop and would be silent scope creep.

## Risks / open questions
- Root cause of the metric-history 400 unknown until reproduced; if ThreadsDash API drift, fix may need endpoint/schema changes (contract: `post_metric_history.read.v1.schema.json`).
- Draft-payload v2 rollout spans two repos: ThreadsDashboard's ingest must deploy dual-accept before exports flip — a cross-repo deploy dependency that needs coordination (v1 is never mutated, so a lagging deploy degrades to rejected v2 exports, loud and recoverable, not corruption).
- Targeted pattern refresh touches `reference_patterns.pattern_json` rows concurrently with any manually-run `analyze_patterns`; last-writer-wins on the same pattern row is acceptable (both embed the same prompt aggregate) but noted.
- New `prompt_post_outcomes` table + derived aggregate changes Reference DB schema (additive migration; existing `outcome_*` columns keep their shape as the aggregate target).
- Hard block at exports.py will surface existing partial-lineage gaps in upstream populators as failures — expected and wanted, but may briefly block exports until populators are patched.
- 50-post cohort pace depends on posting cadence — plan makes the pipe flow, cannot make posts happen.
- Label weight values (1.5/0.9/0.3) are unvalidated priors; retune after first cohort.

- Controlled experiments framework, exploration quotas (audit step 5).
- Autonomy expansion: scheduling, autoposting, orchestrator config (audit step 6).
- Backfilling lineage onto historic posts.
- Unifying Reel winner_score with rewardScore.
- New label-metadata fields (existing `tags` suffices).
- Bulk generation spend.
