# Creator OS — Pipeline Audit

**Date:** 2026-06-16
**Scope:** End-to-end audit of the Creator OS pipeline across all repos (reel_factory, reference_factory, campaign_factory, contentforge, ThreadsDashboard, pipeline_contracts, creator-os).
**Stance:** Pessimistic, evidence-based. Assume systems fail before they succeed.
**Audience:** Codex / implementation. Every finding carries file:line evidence and a concrete fix.

> One-line verdict: the **machine half** (dispatch, metrics ingestion, safety gates, token health, approval enforcement) is genuinely strong (7–8.5/10). The **content-integrity half** (human review, QC/identity, cross-account uniqueness, learning-loop closure, cost governance) is weak (2–4.5/10). Every low score is a content-trust or governance gap, not an infrastructure gap.

## Top risks (ranked by probability × impact)

Impact: critical.** Mitigant: TD repo is **private** (blast radius = repo-access holders, not open internet).
2. **Live `TD_API_KEY` committed plaintext in tracked `ThreadsDashboard/.mcp.json`** (`juno_ak_…`, real production key). Probability: certain. Impact: high.
3. **No spend cap on the active Higgsfield/Kling generation path** (`reel_factory/generate_assets.py` — no budget/balance/max-gen guard; spend controls exist only in the dormant `overnight_grid_worker.py`). A loop bug or oversized batch = unbounded paid-API spend, unmonitored. Probability: medium-high at scale. Impact: high (money).
4. **Byte-identical cross-account reposting of the VIDEO is unguarded.** Camera-variation seed is `camera|recipe|src_hash` (per-source, NOT per-account; `reel_factory/graph_builder.py:87-120`). Ledger dedup keys on `account_id AND content_fingerprint` (`campaign_factory/.../posting_ledger.py:580`) → does not block the same fingerprint across different accounts. **Important nuance:** the autonomous operator's `accountDna.ts` DOES enforce cross-account (sibling) differentiation — but only on the **caption/hook/text** axis (`siblingCollision`, `recentSiblingRepetition`, regenerate on collision, `:1964-2027`). It does NOT fingerprint the media file across accounts. So a reel can get a uniquely-regenerated caption per account while shipping the same byte-identical video to all of them; IG copy-detection scores the video. Also, accountDna guards the auto-post path only — the current manual assignment flow likely bypasses it entirely. Cross-account duplicate VIDEO = fastest farm-ban vector. Probability: high at scale. Impact: high. Fix: per-account render seed (`camera|recipe|src_hash|account_id`) + cross-account media perceptual-fingerprint dedup at assignment.
5. **No ingest boundary.** Campaign Factory writes drafts directly to Supabase `posts` table via service-role key (`campaign_factory/.../adapters/threadsdash.py` `_insert_draft_post`), bypassing ThreadsDashboard validation. TD's envelope validator `validateCampaignFactoryDraftPayload` (`pipeline_contracts/typescript.ts:807`) is test-only/dead at runtime; only per-post publish-time checks run (`publishPreflight.ts`). Batch/v2-manifest violations enter prod DB silently. Probability: medium. Impact: high.
6. **Approval-gate producer is unversioned; review decisions are volatile.** The swipe deck stores decisions in browser `localStorage` (`reel_factory/output/.../*_swipe_review.html:60,71`); persistence depends on a manual "Download JSON" click. The tool that materializes `approved_still_manifest.json` + the `approved/` copies + `approved_sheet.jpg` is **not in any committed source**. Browser clear / wrong machine = total loss; folders silently desync from decisions. Probability: high. Impact: high.
7. **No identity verification of generated faces.** No face-embedding / ArcFace / soul-match anywhere (grep clean across all repos). `ai_visual_qc.py` only counts faces — a wrong/off-model/deformed-single-face render passes QC clean. Identity integrity rests entirely on the fragile manual swipe (risk #6). Probability: medium-high at scale. Impact: high (identity is the product).
8. **Silent burned→IG caption fallback (3 layers)** publishes baity/DM-bait overlay text verbatim as IG caption when no explicit IG caption is set (`campaign_factory/.../adapters/threadsdash.py:1501-1502`, `core.py:25945-25946`, `ThreadsDashboard/api/_lib/publishPreflight.ts:680`). Off-platform language is caught by the discoverability gate, but clean-but-wrong-register text leaks. Probability: medium-high. Impact: medium.
9. **50-account blocker misdiagnosed.** The only inventory gate is sized for 25 (`required = 25 × 3 × 3 = 225`, `campaign_factory/.../core.py:8064`) and PASSES (270 schedule-safe ≥ 225, shortfall 0). The code's own scale analysis names `publishability_pass` throughput as the 50 ceiling (`core.py:9973` tier list). Decisions are being made against the wrong metric. Probability: certain. ## Area detail

### 1. End-to-end architecture (6.0)

Flow as it actually exists: reference intake → Higgsfield direct-reference still (real subprocess) → Stacey Soul ID → 3:4 single still → manual swipe review (localStorage) → approved manifest/folder (untracked producer) → **hard stop** before Campaign Factory (`reel_factory/CURRENT_PRODUCTION_FLOW.md:98`) → (later) Campaign Factory registration → ContentForge variant render → ThreadsDashboard scheduling/publish (real Vercel crons `*/5 * * * *`) → metrics sync (real) → learning summary (report-only).

- **Kling video is a manual handoff**, not automated (`reel_factory/reel_motion_prompt.py`, README:280).
- **Contracts are decorative at ~half the boundaries.** Enforced at publish-time per-post (`publishPreflight.ts`); dead at the producer→DB ingest (risk #5).
- **No ingest boundary** (risk #5).
- `CLOSED_LOOP_PROOF.json` is a real run with all-zero metrics — proves reconciliation/plumbing, not performance lift.

Fix priorities: add a runtime-validated ingest boundary; pick one source of truth (promote or delete the monorepo) and kill the contract drift.

### 2. Generation path (6.0)

- Higgsfield direct-reference is the active wired path (`reel_factory/generate_assets.py:866-868,1234-1244`); Soul ID via `--custom_reference_id` (Stacey `d63ea9c7-…`).
- **Doc/code contradiction:** code defaults 3:4 (`generate_assets.py:22 DEFAULT_DIRECT_REFERENCE_IMAGE_ASPECT_RATIO = "3:4"`, enforced by test), but `CURRENT_PRODUCTION_FLOW.md:41`, `DECISIONS.md:206`, `PIPELINE_BOUNDARIES.md` say 9:16 and list it in "must not break." Fix the docs before an agent "restores" 9:16.
- **Deprecated paths still reachable** (risk #10). Guard with a feature flag that raises on call, or delete.
- **No native still→MP4 motion path.** No `zoompan`/Ken Burns anywhere; all motion → external Kling. `slideshow_factory.py` is static-hold only. `audio_mux.py` works but `03_audio_library/` is **empty** → `select_audio` raises `FileNotFoundError` at runtime (dead muxer). Add ffmpeg `zoompan`/still-hold + populate the audio library if a native path is wanted; otherwise remove the dead muxer.

### 3. Review workflow (2.0)

Two disjoint systems: (A) the **swipe deck** (localStorage HTML, manual export, untracked producer — risk #6); (B) SQLite `review_state` for rendered variations (`reel_factory/manifest.py:364 set_review_state`, consumed by `export_approved.py:53`) with states `{draft, approved, rejected}` — **"maybe" is silently treated as rejected**. The two use incompatible vocabularies and nothing bridges them.

**Recommendation (consolidation, not hardening):** fold the stills swipe flow into the SQLite `manifest.py` model. Extend it with `maybe` + reviewer identity + a decision-history table; make decisions server-persisted; replace the one-way `approved/` copy with a manifest-derived view so folders can't desync; commit the producer to git. Result: one audited, resumable, multi-reviewer review system feeding the still-gated Campaign Factory.

### 4. Caption system (6.0)

- Fields are modeled separately end-to-end: `instagram_post_caption`/`instagram_post_caption_hash` vs `burned_caption_text`/`burned_caption_hash` (`pipeline_contracts/typescript/index.ts:69-75`); export emits both (`campaign_factory/.../adapters/threadsdash.py:146-151`); published `content` = the IG caption, not the burned text.
- **Silent burned→IG fallback (risk #8)** undermines the separation. Kill it at all three sites; require an explicit IG caption or block.
- **Desired baity #fyp style not implemented.** Existing banks are OnlyFans-funnel hooks (DM/comment bait, puzzles), not lightweight "i meannnn #fyp" / "no cappp #fyp" one-liners. Add a reel-post bank with the requested register, and keep regular-post captions normal/simple.
- Caption-outcome tracking (`CAPTION_OUTCOME_TRACKING_V1`) is real plumbing but `manualReviewOnly: True` (`campaign_factory/.../core.py:24799`); the PROOF.json is theater (no test loads it).

### 5. Image-to-MP4 (recommend native, conditional)

See §2. Gap is real (no native motion; dead audio muxer). Add ffmpeg `zoompan`/still-hold + audio mux and populate `03_audio_library/`, OR keep delegating to Kling. Do not ship the half-built native path with an empty audio library.

### 7. Scale readiness (4.5)

- 25 is real and certified by passing live data (270 ≥ 225, shortfall 0).
- **Before 50:** fix the real bottleneck (`publishability_pass` throughput, not inventory — risk #9); add inventory reservation/depletion (net not gross, `SELECT FOR UPDATE` + `reserved_by`); rotate leaked `CRON_SECRET`.
- **Before 100:** raise the reconciliation drain ceiling (cron backstop processes LIMIT 25 stranded items / 5 min ≈ 300/hr; 200 accts × 3 = 600/day backlogs faster than recovery during a QStash outage). Threads fallback caps at `MAX_CONCURRENT_ACCOUNTS=5`.
- **Before 200:** address SPOFs — single Supabase (service-role used by all crons), single QStash, single Upstash Redis; no pgbouncer/connection-pool evidence. Validate horizontal QStash throughput + DB connections under load. Load-test each tier; do not assert.

### ContentForge app (5.5)

Substantial Next.js app (30+ API routes) — the local pre-publish quality-audit + variant tool ("audit in ContentForge" step). Variant Pack flow (`subtle`/`balanced`/`strong`).

- **Variant-pack job runner** (`contentforge/lib/variant-pack-jobs.js`): real idempotency (`sha256` of stable-stringified inputs → runId, `:78-101`); 30-min timeout → `timed_out`; crash-orphan detection (non-terminal + not in in-memory `activeJobs` after restart → `aborted`, `:98-100`).
- **Risks:** state is flat JSON files on local disk (no DB); `writeJob` is plain `writeFileSync` — **non-atomic**, crash mid-write corrupts the file → `JSON.parse` throws on next read. **No concurrency cap** — `startVariantPackJob` fires the runner immediately (no semaphore; reel_factory has a 3-worker cap, this doesn't). **No resume** — restart/deploy mid-run loses in-flight packs. Fire-and-forget runner (`:121-123`) is fine for the local long-lived Next.js process (README: "runs on your machine") but would break on serverless.
- Originality/similarity gates are real and threshold-scored (`lib/campaign-originality-audit.js:326,362`).
- Fix: atomic writes (temp + rename); a concurrency semaphore; optional DB-backed job state if this ever leaves the local box.

### Production engine / variation / spoofing (4.5)

- **Variation** — real seeded ffmpeg fan-out, `N = hooks × recipes × ratios`, 10 default recipes → 10–288 variants/source. Axes: trim, speed, zoom, tilt (±5°), hflip, reverse, EQ, color presets, caption styling, ratio. Quality gate `sscd_check.py` (pHash/dHash/wHash Hamming).
- **Reel assembly** — ffmpeg (`graph_builder.py`), `h264_videotoolbox` + libx264 fallback, captions PIL→PNG→overlay, **3 concurrent workers** → low tens/hour, single machine. Bottleneck = H.264 encode behind the 3-wide semaphore. Horizontal scale needed for 100+.
- **Spoofing — two tiers, wrong target.** reel_factory (mild, wired): seeded crop 95–97%, rotate 0.2–0.5°, hue ±3, colorbalance jitter, pixel noise, `-map_metadata -1` (`graph_builder.py:87-120,272`). contentforge (aggressive, NOT confirmed wired to the Graph-API publish path): PDQ crop-shift, SSCD texture-bias, luma-wave, audio pitch/tempo shift, fake device encoder profiles, fake capture filenames + aged creation_time, JPEG QT randomization. **Two killers:** (a) publishing is via the official **Meta Graph API** (OAuth'd business accounts), so the real farm-ban vector is **account-graph correlation** (shared app / Business Manager / admin / token-refresh IP) — untouched by any pixel work; (b) crop/rotate/hue/noise are exactly the augmentations Meta's learned SSCD detector is trained to ignore, and `sscd_check.py` checks Hamming hashes, not Meta's descriptor — passing the internal gate ≠ evading IG.
- **Cross-account uniqueness — risk #4.**
- Note: `publishFingerprint.ts`/`posting_ledger.py content_fingerprint`/`forensics_check.py` are **self-dedup** (prevent the system reposting itself) — the opposite direction from anti-detection. Do not confuse the two.

**EMPIRICAL VERIFICATION (2026-06-16, executed, not inferred):**
- reel_factory works: 292 tests pass; 589 mp4s + 821 pngs actually produced in `output/` (dated current).
- The camera-variation pixel transform genuinely applies (crop/rotate/hue/noise vary by seed input).
- **Risk #4 confirmed by render test:** `_camera_variation_pre_scale(recipe_name, src_hash)` takes no account input. Rendering clip_027 twice with the same recipe+source produced **byte-identical files** (both `sha256=1d2a8a08ba90d619`, `cmp` identical). Two accounts on the same recipe+source ship the exact same reel.
- **Metadata strip is incomplete (new finding):** even with explicit `-map_metadata -1`, every fresh render retains `encoder=Lavf62.12.100`, `encoder=Lavc62.28.100 libx264`, `handler_name=VideoHandler`, `major_brand=isom`. `-map_metadata -1` strips source tags but not muxer/encoder-stamped ones, so every output carries an ffmpeg/libx264 automation fingerprint (a real phone capture would show an Apple encoder + `creation_time` + device model). Fix: route through the device-profile re-mux, or post-strip/rewrite encoder + handler tags after render.
- **Verdict:** spoofing mechanically functions (pixels change) but fails its purpose — it neither makes per-account copies unique (byte-identical) nor makes output look phone-captured (ffmpeg fingerprint survives). Cosmetic for a multi-account farm.

## Recommended tooling (off-the-shelf, mapped to gaps)

Each tool maps to a specific audited gap. Install + wire-in below. None replaces the design decisions; they close the mechanical holes.

### Spoofing / per-account uniqueness (risk #4 — byte-identical renders + ffmpeg fingerprint)

The empirical test proved two renders of the same recipe+source produce byte-identical MP4s (`sha256` match), and `-map_metadata -1` still leaves `encoder=Lavf62.12.100`, `encoder=Lavc62.28.100 libx264`, `handler_name=VideoHandler` — a stable ffmpeg-automation signature.

- **`exiftool`** (`brew install exiftool`) — strip/rewrite container + stream tags the ffmpeg flag misses. Run as a post-render pass: `exiftool -all= -overwrite_original out.mp4` then re-tag with plausible device metadata. Closes the automation-fingerprint half.
- **`PySceneDetect`** (`pip install scenedetect`) or **`videohash`** (`pip install videohash`) — compute a perceptual hash of the *final* rendered clip and assert cross-account divergence in a test gate. Nothing currently measures whether two accounts' clips actually differ; this makes risk #4 testable instead of assumed.
- Design dependency (no tool fixes it): add `account_id` to the camera-variation seed in `graph_builder.py:_camera_variation_pre_scale` (currently no `account_id` param) so divergence is deterministic-per-account, not random.

### Identity verification (faces counted, not verified as Stacey)

`ai_visual_qc.py` uses YuNet to *count* faces and flag count variance, but never checks the face is the intended creator.

- **`insightface`** (`pip install insightface onnxruntime`, model `buffalo_l`/ArcFace) — extract a face embedding per sampled frame, cosine-sim against a stored Stacey reference embedding; hard-gate below threshold before the swipe queue. Highest-value missing guardrail. Reuses the frame-extraction already in `ai_visual_qc.py`.
- **`deepface`** (`pip install deepface`) — lighter alternative, same cosine-match job, if onnxruntime footprint is a problem.

### Secrets (TD_API_KEY tracked in `.mcp.json`; `.env.production` recoverable in history at commit `c42e8bfb9`)

- **`gitleaks`** — already installed, NOT wired. Add as a pre-commit hook AND a CI step; add a custom rule for the `juno_ak_` key format. Fails the build on any future leak.
- **`git-filter-repo`** (`brew install git-filter-repo`) — one-time purge of `.env.production` and the tracked key from history. (BFG is the alternative.) Rotate every leaked secret first; purge does not un-leak an exposed key.
- **`pre-commit`** (`pip install pre-commit`) — orchestrate gitleaks + ruff + import-linter on commit so the guards run before CI, not only in it.

### Cost ceiling (no spend cap on active `generate_assets.py` path — risk #3)

- No off-shelf tool. Use the **Higgsfield `balance` API/MCP tool** (already available this session) as a precondition before each generation run; hard-stop under a daily-$ + per-run-max-assets threshold. Code, not plugin — needs the dollar numbers from you.

### Job-state durability (`contentforge/lib/variant-pack-jobs.js` — flat-file, non-atomic write, no concurrency cap, no resume)

- **`p-queue`** (`npm i p-queue`) — concurrency cap + retry around the variant runner; drop-in for the fire-and-forget loop (lines 121-123).
- Better: move job state from flat files onto a **Supabase row + Postgres advisory lock** — Supabase is already a dependency; kills the non-atomic `writeFileSync` (line 58) and gives crash-resume for free. No new dependency.

### Inventory reservation race (risk #1)

- Native **Postgres `SELECT … FOR UPDATE`** / advisory locks via Supabase — no plugin. Pair with a `reserved_by` + `reserved_at` column and a TTL release. (See directional decision #1.)

### Observability / CI (broad maturity gap)

Correction to an earlier draft of this audit that flagged "no error tracking" — that was wrong; TD has Sentry. Remaining work is verification, not install: confirm release-identifier env, source-map upload envs, and tunnel/project settings are documented and active in prod. Do NOT add a second integration.
